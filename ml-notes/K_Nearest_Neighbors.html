Space and Time Complexity:¶

Training Time(i.e computing distances not actual training) and Space = O( no_data_pts * dim)

This is a huge setback for this algorithm !!! as it needs high configured hardware and large amount of time.

Theoretical Improvements,

  • Pre-sort training data into fast data structures(like Hash Tables)

  • Compute only an approximate distance

  • Remove redundant data (condensing)

Choice of Nearest Neighbors Algorithm:¶

Query time or prediction time of knn for different algo's are like,

-> Brute force :

     O( no_data_pts * dim )

-> Ball tree :

    O( log(no_data_pts) * dim)

-> kd-tree :

    for dim <= 20, O( log(no_data_pts) * dim) 

    else, O( no_data_pts * dim)

    in this case computations are done internally with dimensionality = 2^dim.

-> in case of Ball tree and kd-tree to get above query times 'K' must be small

-> Brute force performs well in case of small no_data_pts and dim.

Choosing the right 'k':¶

-> First diagram is case of overfit with k = 1 or a smaller k which isn't suited for out training data and with this we give importance to noise which is not good.

-> When k = n or k is very near to n we underfit .

-> To choose right 'k' we will use cross validation

Weighted KNN:¶

-> above for k = 3 everything is alright

-> lets just say that we found k = 5 is suited for entire data then most likely points like that star are going to be misclassified.

-> solution is to give weights , one way is based on distances.

to_decide_class = []

while i < k:

   dist = 1/d(query-pt , i-th-pt)

   append {dist , class of i-th-pt} to 'to_decide_class'

now we'll have 'k' weights(here it's based on distance)

then do 'to_decide_class.group_by(class_label)'

add up all dist for each class label 

finally, declare the class with highest weight as class label of query-pt
